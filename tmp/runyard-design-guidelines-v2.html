Runyard Design Guidelines v2

A reference for AI agents building the Runyard IDE interface. Detailed. No fluff.

1. Philosophy

Runyard is a professional IDE first. AI features are transparent, inspectable operations that produce verifiable results. Not bolted-on magic.

Core Principles

  1. Show the work, not the sparkles. Agent actions display exact logs, files inspected, tools called, outputs produced. Uncertainty is first-class data.
  2. Receipts over theater. The most honest AI interface is an incident timeline: timestamps, actions, evidence, owner, rollback path.
  3. The tool is the hero. Editor, terminal, file tree, git panel are primary. AI features exist within these surfaces.
  4. Silence is signal. A deterministic tool that works needs no commentary. A file that saves successfully needs no toast.
  5. Cross-platform consistency. Desktop, web, mobile share the same visual system. Platform differences handled by the sub-service abstraction.

Contrast Requirements (APCA, Not WCAG Luminance)

  • Body text: Lc 90 minimum
  • Secondary text: Lc 60 minimum
  • Tertiary/disabled: Lc 45 minimum
  • Border text: Lc 30 minimum (structural only, never readable)
  • Focus indicators: Lc 75 minimum with 2px outline

Never use saturated colors for text (red on black, bright green on white). APCA accounts for spatial frequency and polarity; traditional WCAG luminance math does not.

Banned Colors (Permanently)

  • Gradient backgrounds on any UI surface
  • Gradient text or gradient borders
  • Blue-to-purple neon glows (#3B82F6 to #6366F1)
  • Backdrop blur / frosted glass (backdrop-filter: blur)
  • Box-shadow glows simulating neon or magical light
  • Opacity-based dimming as primary state indicator (use explicit tokens)
  • Pure white cards on gray-50 backgrounds
  • High-saturation accent colors for backgrounds (use muted, desaturated variants)

Text Formatting Rules

  • Never use em dashes. Use en dashes for ranges (e.g., 1-10), hyphens for compound words, or periods for independent clauses.
  • Never use emojis in UI text. Use lucide-svelte icons. Text is for reading, not decoration.
  • Never use exclamation marks. Errors are statements. Warnings are statements. Success is silent.
  • Never use conversational filler. "I would be happy to help you with that code today!" is unacceptable. "File saved." is acceptable. "3 tests passed." is ideal.
  • Never use title case for UI labels. Use sentence case: "Open file", not "Open File". "New terminal", not "New Terminal".
  • Capitalize proper nouns only. "Git status", not "Git Status". "Settings panel", not "Settings Panel".

6. The Chat Panel: Specific Rules

The chat panel is the area most vulnerable to AI interface tropes. It must be treated as a first-class tab, equal to the editor and terminal.

6.1 Message Display

Never use chat bubbles. Use a flat message stream:

Message container: full width, no max-width, no border-radius
Message padding: 16px vertical, 0 horizontal (edge-to-edge within panel)
Message separator: 1px solid var(--border)
Message background: none (alternating subtle var(--bg-secondary))
User message: left accent bar 2px solid var(--accent)
Agent message: no accent bar

Never use avatars or profile pictures. Messages are identified by a small, 11px label above the content: "You" or "Claude" or "Agent: code-review". No circular user icons, no robot icons, no initials in colored circles.

6.2 Code Blocks in Messages

Code blocks in chat use the same rendering as the editor:

Container: var(--editor-bg), 1px solid var(--border), 0px radius
Header: 11px, var(--text-secondary), background var(--bg-secondary), padding 4px 8px
Header content: language icon (lucide), filename, copy button, insert button, diff button
Code: JetBrains Mono, 13px, syntax highlighted
Max-height: 400px with scroll, never collapsed by default

Code Block Rules:

  • Always show full filename, never truncated.
  • The copy button is a small icon (14px) in the header, not a floating overlay.
  • "Insert at cursor" or "Replace selection", not "Apply" or "Use".
  • "Show diff", not "Review changes".
  • Never use "Copy to clipboard" toast confirmations. The button briefly shows a checkmark icon for 800ms, then returns to copy icon.

6.3 Diff Cards in Chat

When the agent proposes file changes, display them as inline diff cards:

Diff card: full width, 1px solid var(--border), 0px radius
Header: filename, path, status badge ("modified", "created", "deleted")
Diff content: side-by-side or unified view, syntax highlighted
Hunk actions: "Accept hunk" / "Reject hunk" per hunk, ghost buttons
Card actions: "Accept all" / "Reject all" at top, secondary buttons

Diff Card Rules:

  • Always show the full file path, never just the filename.
  • Green additions: background #1e3a2f (dark) / #dcfce7 (light). Never pure green text on transparent.
  • Red deletions: background #3f1e2f (dark) / #fee2e2 (light). Never pure red text on transparent.
  • Hunk headers (@@ -1,5 +1,7 @@) use 11px, var(--text-tertiary), background var(--bg-tertiary).

6.4 Tool Calls in Chat

When the agent invokes a tool, show a collapsible operation card:

Tool card: full width, 1px solid var(--border), 0px radius
Header: tool icon (lucide), tool name, arguments summary, expand/collapse toggle
Collapsed state: one line, 11px, var(--text-tertiary)
Expanded state: full JSON output, syntax highlighted, 12px mono
Status: running (yellow dot), completed (green dot), failed (red dot)

Tool Card Rules:

  • Tool name displayed as plain text. No sparkles or magic indicators.
  • Running state: small yellow dot (6px, not a spinner), not animated pulse or glow.
  • Completed state: green dot. No checkmark animation. No success message.
  • Failed state: red dot and the error message in full. No "Oops!" or "Something went wrong." Just the error.

6.5 Thinking Blocks in Chat

When the model is in "thinking" or "extended thinking" mode, show a minimal state indicator:

Thinking indicator: 11px, var(--text-tertiary), "Thinking..." with a static ellipsis
Thinking content: collapsed by default, expandable to show raw reasoning tokens
Token count: shown in header when expanded, "~1,247 reasoning tokens"

Thinking Block Rules:

  • Never use animated dots. The ellipsis is static text.
  • Never use a "thinking" animation, glow, or shimmer.
  • Never show thinking content expanded by default. It is collapsed and can be expanded if the user is debugging.
  • The token count is factual data, not a performance brag.

6.6 Permission Requests in Chat

When the agent requests permission (HIL - Human-in-the-Loop), show an inline decision card:

Permission card: 1px solid var(--accent-warning), 0px radius, subtle amber background tint
Header: "Permission required" + operation description
Body: exact tool name, exact arguments, affected files list
Actions: "Approve" (primary), "Deny" (secondary), "Approve for session" (ghost)

Permission Card Rules:

  • Operation description is a dry statement: "Execute shell command: rm -rf node_modules", not "I'd like to clean up your dependencies for you!"
  • Affected files list is a bulleted list of absolute paths, not a summary.
  • "Approve for session" remembers the approval for this tool + these exact arguments within the current conversation. It is a checkbox, not a default.
  • Never use amber/gold glow or border animation. The border is a static 1px line.

6.7 Streaming Text in Chat

Text streams character by character from the agent:

Stream state: text appears in real-time, no typing animation
Cursor: none (the text simply appears, no blinking cursor follows the stream)
Stop button: visible during streaming, ghost button "Stop generation", positioned at the bottom of the message
Regenerate button: visible after completion, ghost button "Regenerate", replaces the stop button

Streaming Rules:

  • Never use a typing animation or a blinking cursor that follows the stream.
  • Never use a "typing" indicator with animated dots.
  • The stop button is functional and interrupts the generation immediately.
  • The regenerate button starts a new generation from the same user message (replaces the previous agent response).

6.8 Context Summary in Chat

When old messages are compressed to save context window, show a divider:

Context summary: dashed horizontal line, 1px, var(--border)
Label: "12 messages summarized" + expand toggle
Expanded: shows the summary text in 12px, var(--text-tertiary), with "View original" link to restore

Context Summary Rules:

  • The divider is a static dashed line, not an animated or glowing element.
  • The summary is factual, not conversational. "12 messages summarized." Not "Let me bring you up to speed..."
  • The original messages are accessible but not expanded by default.

7. Status and Feedback

7.1 Loading States

Skeleton: block-shaped placeholder using var(--bg-tertiary), no animation, no shimmer, no gradient
Skeleton pulse: subtle opacity 0.6 -> 0.8, 1.5s ease, infinite (only for initial load, never for subsequent updates)
Progress bar: 2px height, var(--accent), 0px radius, at top of panel
Spinner: 16px, 1px stroke, var(--text-tertiary), no color animation, no gradient stroke

Loading Rules:

  • Never use skeleton screens with animated shimmer gradients.
  • Never use "spinner with gradient stroke" (the blue-to-purple spinning arc).
  • Never use "loading" text with animated dots.
  • The terminal panel shows "Starting terminal..." as static text during initialization, then removes it when ready.
  • The editor panel shows a 2px progress bar at the top during file load, then removes it.

7.2 Error States

Error banner: full width, 1px solid var(--border-error), background var(--accent-danger) at 5% opacity
Error text: var(--text-error), 13px, --font-sans
Error icon: lucide alert-triangle, 14px, var(--text-error), not animated
Error actions: "Retry" (ghost), "Dismiss" (ghost)

Error Rules:

  • Error messages are statements, not apologies. "Connection failed." not "Oops! We couldn't connect."
  • Error messages include the exact error code or exception: "Connection failed: ECONNREFUSED localhost:7820." not "Something went wrong."
  • Never use red exclamation icons. Never use red pulsing borders. Never use red glow.
  • Never use "Try again later" as the only action. Always provide a retry or a diagnostic step.

7.3 Success States

Success is silent. A file that saves successfully produces no notification, no toast, no checkmark animation. The only indication is the dirty dot disappearing from the tab.

When success absolutely must be communicated (e.g., a git commit that changed remote state), use:

Success indicator: 11px, var(--text-success), static text
"Committed: 3 files changed, 2 insertions(+), 1 deletion(-)"
Duration: 5 seconds, then fade to nothing. No manual dismiss needed.

Success Rules:

  • Never use green checkmark animations. Never use "Success!" messages. Never use confetti or celebratory language.
  • Git operations show a one-line summary in the status bar, not a toast.

7.4 Status Bar

Height: 24px, fixed
Background: var(--statusbar-bg)
Text: var(--statusbar-text), 11px, --font-sans
Padding: 0 8px
Sections: left (connection, git branch, file path, LSP badges), right (cursor position, encoding, language)

Status Bar Rules:

  • Connection status is text only: "Local", "Connected: dev-server", "Connecting...", "Disconnected". Never use colored dots as the sole indicator.
  • Git branch is plain text, preceded by the branch icon (lucide). No color, no animation.
  • LSP badges show language abbreviation (e.g., "TS") with a small colored dot: green (ready), yellow (starting), red (error). The dot is 6px, static, not pulsing.
  • The file path truncates from the left with ellipsis after 50 characters: "...src/components/EditorPanel.svelte".
  • Cursor position is "line:col" format, e.g., "42:15". No "Ln 42, Col 15" verbose format.
  • Never use the status bar for notifications, toasts, or temporary messages.

8. Animation and Motion

8.1 Principles

  1. Motion is functional, not decorative. Every animation must communicate state change, spatial relationship, or feedback. If removing the animation does not reduce comprehension, remove the animation.
  2. Fast is default. Most transitions are 100ms. Complex layout changes (split pane resize, sidebar collapse) are 200ms. Nothing exceeds 300ms except modal fade-in (150ms) and modal fade-out (100ms).
  3. No bouncy or elastic easing. Use ease-out for appearing, ease-in for disappearing, ease-in-out for resizing. Never use spring, bounce, or elastic easing.

8.3 Forbidden Animations

  • Gradient animations (animated gradient backgrounds, gradient text shifts)
  • Glow or pulse animations (pulsing borders, pulsing dots, glowing shadows)
  • Shimmer or skeleton wave effects
  • Bounce or spring physics (elastic resizing, bouncing modals, springy panels)
  • Parallax scrolling or depth effects
  • 3D transforms (perspective, rotateX, rotateY)
  • Blur transitions (fading between blurred and sharp states)
  • Particle effects, confetti, or celebratory animations
  • Typing animations (text that appears character-by-character with a cursor)
  • Animated progress spinners with gradient strokes
  • Morphing shapes or path animations (SVG morphing)

Icon Rules

  • Never use colored icons. All icons are monochrome. The color is determined by the text color of the surrounding element.
  • Never use filled icons. All icons are outline (stroke-based). Filled icons are for high-density mobile contexts only.
  • Never use icon animations. Icons are static. The only exception is the chevron for expand/collapse, which rotates 90 degrees with 100ms transition.
  • Never use the sparkles icon (or any sparkle variant) for AI features. Use bot, cpu, zap, terminal, or code icons to represent AI operations. The sparkles icon is ambiguous and does not communicate "AI" to users (Nielsen Norman Group research).
  • Never use the magic wand icon for AI features. It implies magic and obscures actual functionality. Use wrench, tool, or settings for configuration, play for execution.
  • Never use the robot icon for AI. It anthropomorphizes the system and creates false expectations. Use terminal, code, or file-code for code operations, search for retrieval operations.
  • Never use emoji as icons. The keyboard icon is a lucide keyboard icon, not a keyboard emoji. The warning icon is a lucide triangle-alert, not a warning emoji.

10. Dark Mode and Light Mode

The application supports three modes: dark, light, and system (follows OS preference). The default is dark. The mode is stored in settings.json and applied via the data-theme attribute on the <html> element.

Mode-Specific Rules

  • Never use prefers-color-scheme media queries directly. All theming is token-based.
  • Never hardcode color values in component CSS. Always use CSS custom properties.
  • Never assume dark mode is the default in code logic. Always read the token.
  • The terminal is always dark, regardless of the application theme. Terminal color is independent.
  • The syntax highlighting theme is CodeMirror's oneDark in dark mode, and a high-contrast light theme (e.g., oneLight or custom) in light mode.
  • The diff viewer uses the same token system but with explicit addition/deletion backgrounds that are mode-aware.
  • The AI chat panel uses the same token system for message backgrounds and text.

Theme Token Coverage

Every color value in the application must be defined by a token. If a new component needs a color not covered by existing tokens, a new token must be added to the global CSS and documented here. No exceptions.

11. Accessibility

11.1 Keyboard Navigation

  • All interactive elements must be keyboard accessible (Tab, Enter, Space, Escape).
  • Focus indicators are visible: 2px solid outline, offset 2px, color var(--border-active). Never use outline: none.
  • The command palette is the primary keyboard-driven interface. All commands are searchable by name and category.
  • Global shortcuts are documented in the command palette and shown inline with their keybinding.
  • Modals trap focus and return focus to the trigger element on close.
  • Skip links are available for jumping to the main editor area and the sidebar.

11.2 Screen Reader Support

  • All icons have aria-label or are hidden from screen readers with aria-hidden="true" when decorative.
  • Panel headers use role="region" with aria-label.
  • Status bar information is available via aria-live="polite" for critical changes.
  • Chat messages use role="log" with aria-live="polite" so screen readers announce new messages.
  • Tool call cards use aria-expanded for collapsible state.
  • Diff hunks use aria-label to describe the change: "Added 5 lines to function getConfig."

11.3 Motion Preferences

  • Respect prefers-reduced-motion. When enabled, all transitions become instant (0ms), and skeleton pulsing stops.
  • The prefers-reduced-motion setting is independent of the theme setting.

14. Language and Tone

14.1 Microcopy Rules

  1. No exclamation marks. Periods end statements. No sentence needs enthusiasm.
  2. No conversational filler. "I will", "Let me", "I'd be happy to", "Just a moment" are all banned. Use "Loading...", "File saved.", "3 tests passed."
  3. No emojis. Icons are for visual communication. Text is for reading. Never mix the two.
  4. No em dashes. Use hyphens for compound words, en dashes for ranges, periods for clause separation.
  5. Sentence case everywhere. "Open file", not "Open File". "New terminal", not "New Terminal".
  6. No contractions. "Do not" instead of "Don't". "Cannot" instead of "Can't". Contractions add noise in a tool interface.
  7. No filler adverbs. "Simply", "just", "easily", "quickly" are banned. If the action is simple, the interface should make it obvious without saying so.
  8. Error messages state facts. "Connection failed: ECONNREFUSED localhost:7820." not "Oops! We couldn't connect to the server. Please try again later."
  9. Success messages are terse. "Committed: 3 files." not "Your commit was successful! Great job!"
  10. Warning messages state consequences. "This will discard 3 unsaved changes." not "Are you sure you want to proceed?"
  11. Button labels are verbs. "Save", "Delete", "Commit", "Push", not "Save Changes" or "Delete File".
  12. Menu labels are verbs or nouns. "Open file", "New terminal", "Git status", not "Open File..." or "New Terminal...".
  13. No ellipsis in labels. "Open file" not "Open file...". Ellipses are for truncated text only.
  14. No "please" or "thank you" in interface copy. The interface is a tool, not a service interaction.
  15. No brand names in generic contexts. "Runyard settings" not "Runyard Settings". "Runyard is loading..." is acceptable for the welcome screen only.

14.2 Error Message Template

All error messages follow this template:

[Operation failed]: [Specific error] [Context] [Action]

Examples:
- "Connection failed: ECONNREFUSED localhost:7820. Check the server is running."
- "File save failed: Permission denied on /path/to/file. Retry with elevated permissions."
- "Git commit failed: Nothing to commit. Stage changes before committing."
- "LSP start failed: rust-analyzer not found in PATH. Install rust-analyzer or set path override."

14.3 Status Message Template

All status messages follow this template:

[State]: [Metric or detail]

Examples:
- "Connected: dev-server (192.168.1.100)"
- "Git: main branch, 3 ahead, 1 behind"
- "LSP: TypeScript ready, Python starting"
- "Terminal: bash, 3 sessions"

15. File and Code Structure

15.1 CSS Architecture

All styles are inline within Svelte components using <style> tags. No external CSS files except the global global.css.

  • Component-level styles use the component's class name as the root selector (e.g., .editor-panel for EditorPanel.svelte).
  • No global CSS selectors in component files. All component selectors are scoped by the component class.
  • CSS custom properties are the only source of color values. Never hardcode hex colors in component CSS.
  • CSS custom properties are defined in global.css only. No component defines new CSS properties.
  • If a new token is needed, add it to global.css and document it in this specification before using it.

15.2 Component File Structure

Each component follows this structure:

<script lang="ts">
  // 1. Imports (type imports first, then value imports)
  // 2. Type definitions (interfaces, types)
  // 3. Props declaration with $props()
  // 4. State with $state()
  // 5. Derived state with $derived()
  // 6. Effects with $effect()
  // 7. Functions (event handlers, helpers)
  // 8. Lifecycle (onMount, onDestroy)
</script>

<!-- Template: minimal DOM, semantic HTML -->

<style>
  /* Scoped styles using component root class */
  .component-name { ... }
  .component-name .child { ... }
</style>

15.3 Naming Conventions

  • Components: PascalCase, descriptive (e.g., EditorPanel.svelte, GitStatusBar.svelte)
  • CSS classes: kebab-case, component-prefixed (e.g., editor-panel, editor-panel-content)
  • CSS custom properties: kebab-case with semantic names (e.g., --bg-secondary, --text-error)
  • Variables: camelCase for TS, snake_case for Rust (e.g., filePath, terminal_id)
  • Event names: event:subject format (e.g., fs:changed, terminal:output, lsp:message)
  • Command IDs: category.action format (e.g., file.open, terminal.new, git.commit)

17. Checklist for New Components

Before shipping any new UI component, verify:

  • All colors use CSS custom properties from the token system, never hardcoded hex values.
  • No gradient backgrounds, gradient text, or gradient borders.
  • No backdrop blur, frosted glass, or glassmorphism effects.
  • No box-shadow glows or neon effects.
  • No border-radius on code, data, or file tree elements (0px radius for structural content).
  • No emojis in any text or label.
  • No em dashes in any text; use hyphens, en dashes, or periods.
  • No exclamation marks in any text.
  • No conversational filler ("I will", "Let me", "Just", "Simply").
  • No sparkles, magic wands, or robot icons for AI features.
  • No animated loading spinners with gradient strokes.
  • No shimmer or skeleton wave animations.
  • No pulsing, breathing, or glowing animations.
  • No bounce, spring, or elastic easing.
  • No 3D transforms, parallax, or depth effects.
  • No chat bubbles for code or structured data.
  • No mock IDE elements, fake terminals, or simulated code.
  • No "Are you sure?" language; state consequences directly.
  • No "Something went wrong" or "Oops!" error messages; show exact errors.
  • All text is sentence case, not title case.
  • All interactive elements are keyboard accessible with visible focus indicators.
  • All icons have aria-label or aria-hidden.
  • All animations respect prefers-reduced-motion.
  • All text meets APCA Lc 90 for body text, Lc 60 for secondary text.
  • Component styles are scoped and do not use global selectors.
  • No new CSS custom properties are defined in component files; all tokens are in global.css.
